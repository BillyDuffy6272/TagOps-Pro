# Seed Data

This folder is where actual TagOps-Pro records live in editable form. Each JSON file holds an array of records for one entity. To add data, open the relevant file and append another object to the array.

This is the bridge between the markdown data dictionary (in `docs/04-data-model.md` and `docs/data-dictionary-examples/`) and a real Supabase database. The JSON here is the source of truth for example/seed data; the SQL schema (`schema.sql`) describes the database structure those records will eventually live in.

> **Status: Week 2 draft.** The shapes here are expected to shift in Weeks 5–6 as the proper data model and migrations are produced. Treat this folder as the "fillable" version of the data dictionary — not a finished seed pipeline.

## How to add data

1. Open the relevant `NN-<entity>.json` file.
2. Copy the last object in the array and paste it as a new array element.
3. Edit the values. Update the `Id` (e.g., the next sequential `USRID_AG_0003`) and any foreign-key references (`OrganisationId`, `ContainerId`, etc.) so they point at existing records.
4. Save. Optional: validate the file is still valid JSON (e.g., `cat 01-users.json | jq .`).

When a new entity is needed (e.g., a brand-new organisation), add records consistently across files so foreign-key references close: a new Organisation needs an owner User and an owner Membership; a new Container needs an Organisation; a new Tag needs a Container; and so on.

## Field naming and ID conventions

JSON keys use CamelCase to match the data dictionary (`UserId`, `OrganisationId`). The SQL schema uses Postgres-convention `snake_case` columns (`user_id`, `organisation_id`). The conversion is trivial at load time.

Display IDs follow `PREFIX_XX_YYYY`:

- `PREFIX` — entity type (`USRID`, `ORGID`, `MEMID`, `CNTID`, `TAGID`, `TRGID`, `VARID`, `CONID`).
- `XX` — two-letter organisation code (e.g., `AG`).
- `YYYY` — zero-padded sequential number per type per organisation.

In JSON, foreign keys reference the **display ID** of the parent record (e.g., `OwnerId: "USRID_AG_0001"`). When this data is eventually loaded into Supabase, a small load script will:

1. Insert each row, letting Postgres generate the UUID primary key.
2. Maintain a map of `display_id → uuid` for the rows just inserted.
3. Resolve any FK fields that currently hold a display ID into the corresponding UUID before inserting the dependent rows.

This keeps the JSON human-readable while still letting the database use UUIDs as actual PKs (Supabase convention, and required to keep IDs unguessable in URLs).

## What's NOT in here

- **Passwords.** Handled exclusively by Supabase Auth — never stored in our own tables (security floor #6).
- **OAuth tokens.** Roadmap feature (`integrations` table); will be stored encrypted via Supabase Vault when implemented.
- **Audit-log rows.** Generated by Postgres triggers at runtime, not seeded.
- **Soft-delete `deleted_at` columns.** Omitted from JSON for clarity — assume `null` unless a record is explicitly marked deleted.

## Validating the data

Each JSON file should:

1. Be valid JSON (parseable by any JSON parser).
2. Have every foreign-key field point at an existing record in another file.
3. Match the field shapes described in `../docs/04-data-model.md`.

A simple validation script can be added in Week 5–6 (e.g., a Node or Python script that loads each file, builds the FK graph, and reports orphans).
